# Data Models

This document defines the core data structures used throughout NagrikAI.

## Primary Entities

### Report
The fundamental entity representing a civic issue. Defined in `src/types.ts`.

| Field | Type | Required | Description |
|---|---|---|---|
| `id` | `string` | Yes | Unique identifier (currently generated via `Date.now().toString()`). |
| `type` | `string` | Yes | Classification by Vision Agent (e.g., 'pothole', 'garbage'). |
| `category` | `string` | No | Broader category (e.g., 'Infrastructure'). |
| `description` | `string` | Yes | Text description, generated by AI or user. |
| `severity` | `number` | Yes | Final adjusted severity (1-5). |
| `originalSeverity`| `number` | No | Initial severity from Vision Agent. |
| `severityReasoning`| `string`| No | Explanation from Severity Agent for the adjustment. |
| `agency` | `string` | Yes | Responsible government body assigned by Jurisdiction Agent. |
| `ward` | `string` | No | Specific city ward. |
| `location` | `string` | Yes | Human-readable address. |
| `lat` | `number` | No | Latitude coordinate. |
| `lng` | `number` | No | Longitude coordinate. |
| `date` | `string` | Yes | ISO 8601 timestamp of creation. |
| `status` | `string` | Yes | Lifecycle state: "Pending Verification", "Community Verified", "In Progress", "Resolved". |
| `verified` | `boolean` | Yes | True if matched with existing reports or voted by community. |
| `costEstimateLow` | `number` | No | Lower bound of repair cost (INR). |
| `costEstimateHigh`| `number` | No | Upper bound of repair cost (INR). |
| `costReasoning` | `string` | No | AI explanation for the cost estimate. |
| `imageUrl` | `string` | No | Base64 or URL of the primary image. |
| `imageUrls` | `string[]` | No | Array of Base64 strings or URLs for all images. |
| `slaDeadline` | `string` | No | ISO 8601 timestamp indicating when the agency must resolve this. |
| `impactScore` | `number` | No | Calculated priority score (0-100) based on severity, duplicates, and verification. |
| `comments` | `Comment[]`| No | Array of user comments. |

### Comment
Represents a community update or discussion on a Report.

| Field | Type | Required | Description |
|---|---|---|---|
| `id` | `string` | Yes | Unique identifier. |
| `author` | `string` | Yes | Name of the user (or 'Anonymous Citizen'). |
| `text` | `string` | Yes | The comment content. |
| `date` | `string` | Yes | ISO 8601 timestamp. |

### ReportEmbedding
Used internally in `server/db.ts` for semantic similarity search.

| Field | Type | Required | Description |
|---|---|---|---|
| `id` | `string` | Yes | Foreign key to `Report.id`. |
| `vector` | `number[]`| Yes | The 768-dimensional float array generated by `text-embedding-004`. |

## Database Persistence (Current State)

The application currently uses an **in-memory** data store located in `server/db.ts`.

```typescript
export const reports: Report[] = [];
export const reportEmbeddings: { id: string; vector: number[] }[] = [];
```

### Migration Path

The presence of `src/db/schema.ts` and `src/db/drizzle.config.ts` indicates an intended migration to a PostgreSQL database using Drizzle ORM. However, the `server.ts` currently imports directly from `server/db.ts` arrays.

To migrate to production:
1. Initialize a PostgreSQL database (e.g., Cloud SQL or Supabase).
2. Ensure the `pgvector` extension is enabled.
3. Update `server.ts` to execute Drizzle ORM inserts/selects instead of array `.push()` and `.find()`.
4. Replace the manual `cosineSimilarity` function in `server/db.ts` with Drizzle's vector operators.
